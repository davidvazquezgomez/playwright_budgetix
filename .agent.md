# Agente: Convertir literales a constantes automáticamente

Objetivo
- Reemplazar literales repetidos usados para roles (p. ej. 'button') por constantes (`ROLE_BUTTON`) y añadir la declaración de la constante al inicio del archivo.

Cuándo ejecutarlo
- Antes de commits o como paso del agente cuando edita tests/features.

Cómo usarlo
1. Ejecuta el script: `node scripts/convert-literals.js` (por defecto actúa sobre `features/**/*.ts`).
2. Para incluir otros directorios, pasa un patrón glob: `node scripts/convert-literals.js "pages/**/*.ts"`.

Reglas que aplica
- Convierte llamadas como `getByRole('button', ...)` a `getByRole(ROLE_BUTTON, ...)` y agrega `const ROLE_BUTTON = 'button';` tras los imports.
- Evita duplicar constantes si ya existen.

Extensiones / mejoras
- Puedes ampliar `ROLE_NAMES` dentro de `scripts/convert-literals.js` para más roles o adaptar el script para convertir selectores (p. ej. `#username`) en constantes.

Instrucción para el agente
- Cuando el agente detecte cambios en archivos de tests/features, ejecutar el script y añadir los cambios al mismo commit. Si hay conflictos, notificar al humano.

Nota para el agente (importante)
- Usuario: QA junior. Ajusta tu salida y cambios pensando en que el receptor es junior.
- Código claro: prioriza claridad y simplicidad sobre trucos concisos; usa nombres descriptivos, pequeñas funciones y constantes legibles.
- Explicaciones: para cada cambio automático, añade un breve comentario en el PR o en el mensaje de commit explicando qué se cambió y por qué.
- Mantén normas: no rompas las reglas de estilo o convenciones existentes del repo (formato, lint, tests, nombres de Page Objects).
- Revisión humana: siempre crea un PR (o un commit con mensaje claro) y no merges automáticos; si el cambio afecta muchos archivos, avisar al equipo.

Cómo actuar en casos dudosos
- Si una sustitución podría aplicarse en contextos distintos (p. ej. el mismo selector usado en un string de documentación), evita el cambio y registra el hallazgo.
- Si el script encuentra literales nuevos no contemplados, crea una entrada en el PR con ejemplos y sugiere la constante apropiada.

Flujo conservador (opción 2)
- Propósito: el agente automatiza el refactor mínimo (ejecuta el script), pero toda explicación final la genera el Explainer manualmente para revisión por un humano.
- Pasos que debe seguir el agente principal cuando haga cambios automáticos:
	1. Ejecutar `node scripts/convert-literals.js` sobre el scope solicitado.
	2. Crear una nueva rama: `agent/convert-literals/<timestamp>`.
	3. Hacer commit de los cambios con un mensaje claro. Ejemplo de mensaje:
		 - `chore(refactor): extraer literales de roles a constantes (ROLE_BUTTON)\n\nFiles: features/steps/Login.steps.ts`.
	4. Abrir un PR usando `/.github/PULL_REQUEST_TEMPLATE.md` y no mergearlo automáticamente.
	5. En el cuerpo del PR, añadir una nota breve pidiendo al agente Explainer o a un humano que complete `EXPLAINER.md` con pasos de verificación y explicaciones para QA junior.
	6. Notificar al equipo (ej. comentar `@team-qa` en el PR) y asignar la tarea de redacción del Explainer.

- Regla adicional: si los cambios afectan más de N archivos (por ejemplo 10), el agente debe parar y avisar a un humano en lugar de abrir el PR automáticamente.

Responsabilidad del Explainer (manual)
- El Explainer (humano o agente) recibe la solicitud y redacta `EXPLAINER.md` en el PR con el formato definido en `.agent-explainer.md`.
- Sólo tras la aprobación del Explainer y revisión humana se procede al merge.
